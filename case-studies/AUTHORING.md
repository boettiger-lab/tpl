# Authoring geo-agent case-study walkthroughs

A guide for adapting the pattern in this directory to other geo-agent apps in
the boettiger-lab stack (`tpl-ca`, `tpl-conservation-almanac`, future apps for
other partners and regions).

These notes assume someone — human or LLM — has been pointed at the three
case studies here and is about to write a fourth.  Reading the examples
captures the *form*.  This document captures the *discipline* that made the
form work.

## The single most important rule

**Anchor every walkthrough to one specific deliverable type, one specific
place, and one specific political question — in that order — before you
write anything else.**  The deliverable type fixes the section list; the
place fixes the data filters; the political question fixes the framing.

Bad opening: "let's tell stories about voter approval rates."
Good opening: "Senator Hooper has a 2024 ask for $300M to the Florida
Forever Trust Fund and we need to put a brief in front of her by Tuesday."

The earlier slide-deck attempt in this repo (`slides/_archived/`) failed
because it skipped this step.  It told true facts in service of no
particular question, anchored to no particular place, mapped to no
particular TPL deliverable.  Don't make that mistake.

## The structural recipe

For each walkthrough:

1. **Read the partner's existing report template.**  In this repo that's
   `example-reports/` — the 1-pager, the 23-page feasibility study, the
   81-page climate-finance study.  Other apps will have analogs: USFS
   forest-plan briefs, a state wildlife action plan, a foundation grant
   report.  Read at least one to the point of being able to list its
   table of contents from memory.

2. **Lock the section list before writing.**  The walkthrough's section
   headings mirror the target report's section headings.  Story C in the
   FL SD 21 case study is `§ Voter support` because the original brief has
   that section, not because the agent has interesting voter data.

3. **For every section, write four parts in order:**
   - **The advocacy or research goal** — one sentence on what the partner
     is trying to communicate.
   - **The natural-language prompt** — the exact text a TPL analyst would
     type into the geo-agent.  Phrased as a question, in plain English.
   - **The app's response** — a real table, a real number, a real map
     screenshot.  Generated by running the prompt's underlying SQL via
     the MCP tool, not invented.
   - **How it lands in the report** — which paragraph, which table cell,
     which bullet point of the partner's deliverable does this fill?

4. **Compare to a known-good source whenever possible.**  The FL SD 21 case
   study works because the brief's $11.4M Wall Springs figure matches the
   app's $11.44M to the cent.  That single match is more credible than any
   amount of new data.  If the partner's report has a number, reproduce it
   first, surface new content second.

## What to put in green callouts, orange callouts, and disclosures

- **Green prompt callout (`<div class="prompt">`)** — the exact text the
  analyst types.  Should read as a question someone would actually ask.
  Never paste raw SQL into a prompt callout.

- **Grey response callout (`<div class="response">`)** — one paragraph
  describing what the app does internally (which dataset, which join).
  Optional; use when the result needs framing.  Don't repeat the SQL.

- **Expandable SQL disclosure (`<details class="sql">`)** — the actual SQL
  the MCP tool runs.  Closed by default; for the reader who wants to
  verify.

- **Orange gap callout (`<div class="gap">`)** — the explicit acknowledgement
  of what the app cannot do for this section, and the external source that
  fills the gap.  **These are load-bearing.**  A walkthrough with no gap
  callouts is dishonest about the tool's limits; partners notice.

The CSS for all four is in `case-studies/assets/walkthrough.css`.

## Compute model

All aggregation runs server-side via the `duckdb-geo` MCP tool.  The walkthrough
HTML contains only **small** result tables and inline JSON for in-page
visualizations.  Do not write a local Python pipeline to compute joins —
the laptop's bandwidth is the bottleneck and the MCP server is not.

**H3 hex SEMI JOIN before aggregate** is the canonical pattern for any
intersection.  See the SQL disclosures in the existing case studies for
the template — small mask hex → SEMI JOIN to global hex parquet → GROUP BY.

## What goes in the rendered page

- **One MapLibre map per case study** loading the partner's existing
  PMTiles layers directly from public S3.  Same URLs the live app uses.
  Highlight the place with a filter expression on the relevant boundary
  layer; label the 5–10 named entities mentioned in the prose with an
  inline GeoJSON.
- **Tables, not charts, for most data.**  A `<table class="t">` styled
  per `walkthrough.css` works for everything from a 4-row equity table
  to a 20-row site list.  Reserve charts for time series or
  scatter — and even then, render them with Observable Plot from inline
  JSON, not as static images.
- **A draft of the final deliverable at the end.**  Styled as
  `<div class="brief">` so it visually evokes the partner's existing
  document.  Items the agent produced go in plain face; items still
  needing external sources are tagged `<span class="tag gap">`.

## What NOT to do

- **Don't lead with aggregate national statistics decoupled from a
  decision.**  "$156B has been approved nationally" is true but useless
  without a Senator Hooper to ask.
- **Don't generate visualizations the partner's existing deliverable
  doesn't use.**  TPL's reports are prose + tables + maps.  An interactive
  scatter chart that no analyst would put in a PDF is decoration.
- **Don't hide the gaps.**  If 60% of the report needs a CAFR, write that
  in section 1 and again in section 8.  Partners trust the tool more when
  it tells them what it can't do.
- **Don't write prose that reads as if the agent itself wrote it.**  The
  walkthrough is a *document about how to use the agent*, not a thing the
  agent emits.  Voice should be a TPL analyst's notebook, not a chatbot's
  reply.
- **Don't commit a half-finished case study to demonstrate that the
  approach scales.**  The discipline is in the depth.  One complete
  case study beats three half-complete ones.

## A practical checklist for the next walkthrough

Before writing the HTML:

- [ ] I have read the partner's existing report end-to-end.
- [ ] I can list its table of contents from memory.
- [ ] I have picked a specific place (county, district, watershed).
- [ ] I have picked a specific political question
      (the ask, the ballot window, the funding decision).
- [ ] I have run at least 5 MCP queries to confirm the place's data
      story is what I think it is.  I have at least one
      `match` (a published partner number my query reproduces) and at
      least two `new` items (content the partner's report omits that
      the agent surfaces).

Before merging:

- [ ] Every figure in the walkthrough ties to a query in a `<details>` block.
- [ ] Every section ends with either a `gap` callout or a stated reason
      the section has no gaps.
- [ ] The draft of the final deliverable at the end of the page is
      something the partner could actually copy-paste into their template.
- [ ] The page is reachable from `case-studies/index.html`.
- [ ] A headless browser walks every named anchor without errors and
      the MapLibre map's canvas mounts.

## File layout for a new app

Copy this directory's structure verbatim:

```
case-studies/
├── index.html                   # cards linking to each case
├── <slug>.html                  # one per case study
├── AUTHORING.md                 # this file (adapt the partner references)
└── assets/
    └── walkthrough.css          # unchanged
```

The CSS in `walkthrough.css` is partner-agnostic; the palette uses
TPL greens but any deployment can override `--tpl-green` and friends
at the top of the file.

## When to add a new geo-agent data layer

If three case studies in a row need the same external source — Census
ACS demographics, county CAFR data, precinct-level election results —
that's the signal to add it as a layer in the STAC catalog and a hex
parquet on `s3-west.nrp-nautilus.io`.  See
`example-reports/02-geo-agent-tasks.md` for the existing wishlist.
Don't paper over the gap with a hand-curated workaround in the
walkthrough; mark it `gap` and add it to the layer wishlist.
